《Linux/UNIX系统编程手册》第5章 文件I/O：更多细节

发表于 2024-10-05 更新于 2024-11-08 分类于 计算机系统 本文字数： 13k

　　所有系统调用都是原子执行的，即内核会保证系统调用在执行时不会被中断，这避免了竞争。竞争是2个进程或线程产生的操作共享资源的结果取决于无法预期的情况 (进程或线程访问CPU的相对顺序)。

　　系统调用fcntl()能够对打开的文件描述符执行多种控制操作。参数cmd用于指定操作。由省略号表示的第3个参数可以是任何类型，或者被忽略，内核根据参数cmd来确定该参数的期望类型 (若存在)。

　　内核维护了打开文件表 (系统范围)，其中的条目称为打开的文件句柄，包括当前文件偏移量、打开文件时指定的状态标志、文件访问模式、信号驱动I/O的相关设置和指向i-node表条目的指针。

　　对于每个进程，内核都为其维护了打开的文件描述符表，简称文件描述符表。该表中的每个条目都记录了1个文件描述符的信息，包括1组用于控制对应文件描述符的操作的标志位和1个指向打开的文件句柄的指针。

　　每个文件系统都有i-node表，为该文件系统中驻留的所有文件所用，其条目包括文件类型和权限、指向该文件持有的锁列表的指针和文件的各种属性 (例如，文件大小和不同类型的文件操作的相关时间戳)。

　　i-node在磁盘上的表示和内存中的表示不同。磁盘上的i-node记录了文件的永久属性 (例如，文件类型、权限和时间戳)。当访问文件时，会创建内存中的i-node副本，该版本的i-node记录了引用i-node的打开的文件句柄的数量以及从中复制i-node的设备的主要ID和次要ID。此外，内存中的i-node还记录了文件打开时与文件关联的临时属性 (例如，文件锁)。

　　如图所示，进程A和B有大量的打开的文件描述符。进程A的描述符1和20引用的打开的文件句柄相同，这种情况可能是因为调用了dup()、dup2()或fcntl()。进程A的描述符2和进程B的描述符2引用的打开的文件句柄相同，这种情况是因为调用了fork()或进程通过UNIX域名套接字传递文件描述符给其他进程。

　　进程A的描述符0和进程B的描述符3引用的打开的文件句柄不同，但这2个句柄引用的是同一i-node表条目 (即同一文件)。这种情况是因为2个进程都对该文件调用了open()。当同一进程多次打开同一文件时，会出现类似的情况。

　　综上所述，引用同一打开的文件句柄的不同文件描述符共享文件偏移量，类似的规则也适用于打开文件状态标志。相反，文件描述符标志仅限于进程和文件描述符，对这些标志的修改不会影响同一进程的其他描述符或其他进程。

5.3 复制文件描述符

　　系统调用dup()返回的新文件描述符 (未使用的编号最小的描述符) 引用的打开的文件句柄与参数oldfd的相同。

　　系统调用dup2()与dup()类似，但返回的文件描述符由参数newfd指定 (即两者相同)。当参数newfd已经打开时，该系统调用会先关闭它，并且关闭时出现的错误会被忽略。因此，应该在调用dup2()前显式地关闭参数newfd指定的文件描述符。若oldfd是无效的文件描述符，则dup2()会失败 (将errno设置为EBADF)，并且不会关闭newfd。若oldfd是有效的文件描述符，并且oldfd和newfd相同，则dup2()什么也不会做 (不会关闭newfd)。

　　对于复制文件描述符，系统调用fcntl()的操作F_DUPFD可以提供更多的灵活性。

　　这种调用方式会使用编号不小于startfd的未使用的文件描述符来复制oldfd。对dup()和dup2()的调用都可以替换为对close()和fcntl()的调用 (有时，dup2()设置errno的错误码和fcntl()的不同)。

　　在5.3中提到过，复制文件描述符后，新文件描述符与原文件描述符共享文件偏移量和打开文件状态标志。但是，新文件描述符有自己的文件描述符标识 (集)，并且close-on-exec标志是关闭的。

　　系统调用dup3()与dup2()执行的任务相同，但添加了参数flags。参数flags是位掩码，用于修改该系统调用的行为。此外，dup3()还支持标志O_CLOEXEC。dup3()是Linux 2.6.27新增的系统调用，并且仅限于Linux。

　　从Linux 2.6.24开始，Linux还支持符系统调用fcntl()的操作F_DUPFD_CLOEXEC。该操作与F_DUPFD的功能相同 (复制文件描述符)，但该操作会为新文件描述符设置close-on-exec标志。SUSv4详细说明了该操作，但SUSv3没有说明该操作。

5.4 指定偏移量处的文件I/O

　　除了能够指定执行文件I/O的位置之外，系统调用pread()和pwrite()与read()和write()的功能相同。此外，pread()和pwrite()不会修改文件偏移量。

　　系统调用pread()和pwrite()在多线程应用程序中很有用，因为进程的所有线程共享文件描述符表 (所以它们也共享文件偏移量)。

　　但是，系统调用readv()和writev()的glibc包装函数会隐式地执行了一些额外的任务。若这些系统调用因iovcnt过大而失败，则包装函数会临时分配1个足以存储iov所有元素的缓冲区，并调用read()或write()。

5.5.1 分散输入和聚集输出

　　系统调用readv()会执行分散输入，即从参数fd引用的文件中读连续的字节序列，然后将这些字节存放到参数iov指定的缓冲区中。iov指定的缓冲区以iov[0]开头。

　　readv()的重要属性之一是以原子方式完成。从调用进程的角度，内核只会在fd引用的文件和用户内存之间进行1次数据传输。因此，当readv()读文件时，读取的字节必定是连续的，即使共享同一文件偏移量的其他进程 (或线程) 试图在此过程中修改偏移量。

　　若readv()正常运行时，则返回实际读取的字节数；若遇到EOF，则返回0。调用者需要根据返回值来确定是否读取了所有请求的数据。若有效数据不足以填充所有缓冲区，则仅会填充部分缓冲区，并且最后1个缓冲区可能只会被填充一部分。

　　系统调用writev()会执行聚集输出，即将参数iov指定的缓冲区中所有数据连接起来 (按数组顺序)，然后将它们作为连续的字节序列写入参数fd引用的文件描述符中。同样，iov指定的缓冲区也以iov[0]开始。

　　与readv()一样，writev()也是以原子方式完成。在用户内存和fd引用的文件描述符之间的数据传输仅会作为1个操作执行，因此，当writev()写数据时，所有请求的数据都会连续写入到文件，不会被其他进程 (或线程) 中断。

　　writev()可能出现不完全写的情况，所以需要检查返回值来确定所有请求的数据是否已写入。

5.5.2 在指定偏移量处执行分散-聚集I/O

　　Linux 2.6.30新增了系统调用preadv()和pwritev()，用于在指定偏移量处执行分散-聚集I/O。这2个系统调用并非标准，但现代BSD中都包含它们。

　　系统调用preadv()和pwritev()执行的任务与readv()和writev()相同，但会在参数offset指定的位置处。

5.6 截断文件

　　系统调用truncate()和ftruncate()会将文件截断到参数length指定的值。若文件大小大于length，则丢弃多余的数据；若文件大小小于lenght，则补上空字节序列或空洞。

　　这2个系统调用的区别在于指定文件的方式。对于truncate()，参数pathname指定的文件必须是可访问和可写的 (若是符号链接，则会解引用)。对于ftruncate()，参数fd是 (至少以只读模式) 打开的文件描述符，该系统调用不会改变文件偏移量。

　　当length大于文件大小时，SUSv3给出了2个选择，即扩展文件 (对于Linux) 或返回错误。对于truncate()，SUSv3要求扩展文件。此外，XSI也要求扩展文件。

　　系统调用truncate()是仅有的可以直接修改文件内容 (无需先通过open()或其他方法来获取文件描述符) 的系统调用。

5.7 非阻塞I/O

　　在打开文件时指定标志O_NONBLOCK有2个目的：

　　● 若文件不能立即打开，则open()会返回错误 (而非阻塞中)。与有名管道一起使用时，open()可以阻塞。

　　● 当成功调用open()后，后续的I/O操作也是非阻塞的。若系统调用不能立即完成，则只会传输部分数据，或者失败，并将errno设置为EAGAIN或EWOULDBLOCK。系统调用出错时设置的errno值取决于系统调用本身。对于Linux和很多UNIX实现，这2个错误常量是同义词。

　　非阻塞模式可以用于设备 (即终端和伪终端)、无名管道、有名管道和套接字。因为无名管道和套接字的文件描述符不是通过open()获得，所以必须通过fcntl()的操作F_SETFL来启用该标志。

　　常规文件会忽略标志O_NONBLOCK，因为内核缓冲区会确保对常规文件的I/O不会阻塞。但是，当使用强制文件锁定时，该标志也会对常规文件生效。

　　从历史角度，System V衍生的实现提供了与标志O_NONBLOCK相似的标志O_NDELAY。主要区别在于，若write()无法完成或没有足够的输入来满足read()时，非阻塞的write()会返回0。这种情况不适用于read()，因为无法区分这种情况与EOF情况。因此，POSIX.1新增了标志O_NONBLOCK。有些UNIX实现仍然提供具有旧语义的标志O_NDELAY。Linux定义了标志O_NDELAY，但它与标志O_NONBLOCK是同义词。

5.8 大文件的I/O

　　用于表示文件偏移量的数据类型off_t通常实现为有符号长整型数 (实现为有符号数是因为值-1表示错误情况)。对于32位架构，这会将文件大小限制为231-1字节 (即2GB)。但是，磁盘驱动程序的容量早就超过了该限制，所以需要32位UNIX实现来处理大于该限制的文件。由于这个问题在很多实现中都很常见，UNIX供应商联盟在大型文件峰会中进行了合作，来让SUSv2支持大文件的访问 (完整的LFS规范于1996年定稿)。

　　从内核2.4开始 (还需要glic 2.2及以上)，Linux提供了32位系统的LFS支持。此外，对应的文件系统也必须支持大文件。大多数Linux原生文件系统都支持大文件，有些非原生文件系统不支持大文件 (Microsoft的VFAT和NSFv2严格限制文件大小为2G，无论是否使用了LFS扩展)。

　　64位架构的长整型数实现为64位，文件大小的理论上限是263-1字节，在大多数情况下，该大小远大于磁盘大小。因此，64位架构通常不需要对文件大小进行限制。

　　可以通过以下2种方式之一来开发需要LFS功能的应用程序：

　　● 使用支持大文件的替代API。LFS将该API设计为SUS的 "过渡扩展"。因此，符合SUSv2或SUSv3的系统并不需要提供此API，但很多符合SUSv2或SUSv3的系统还是提供了此API。这种方法现已过时。

　　● 编译程序时将宏_FILE_OFFSET_BITS定义为值64。这是首选方法，因为它允许符合规范的应用程序在不修改源码的情况下获取LFS功能。

　　若通过32位函数来访问大文件 (不使用LFS功能)，则可能会遇到错误EOVERFLOW。例如，当使用函数stat()的32位版本来获取大于2GB的文件的信息时，就会出现该错误。

5.8.1 过渡LFS API

　　为了使用过渡LFS API，需要定义功能测试宏_LARGEFILE64_SOURCE。该API提供的函数足以处理64位的文件大小和偏移量，其名称与对应的32位函数类似 (使用了后缀64)，包括fopen64()、open64()、lseek64()、truncate64()、stat64()、mmap64()、setrlimit64()。

　　为了打开大文件，可以直接使用这些函数的64位版本。

　　调用open64()等同于调用open()时指定标志O_LARGEFILE。若通过open()打开大于2GB的文件时不指定标志O_LARGEFILE，则会出错。

　　除了上述函数，过渡LFS API还提供了一些新数据类型，包括struct stat64和off64_t。前者类似于结构体stat，但支持大文件。后者是表示偏移量的64位数据类型，用于函数lseek64()。

5.8.2 宏_FILE_OFFSET_BITS

　　定义宏_FILE_OFFSET_BITS的方式有2种，即在C源码中定义#define _FILE_OFFSET_BITS 64 (引入头文件之前) 或对C编译器使用命令行选项-D_FILE_OFFSET_BITS=64。

　　宏_FILE_OFFSET_BITS会将所有相关的32位函数和数据类型转换为对应的64位版本。例如，对open()的调用会转换为对open64()的调用，数据类型off_t会定义为64位。因此，若程序仅使用数据类型为off_t的变量来表示文件偏移量，则只需要重新编译程序就可以支持大文件。

　　宏_FILE_OFFSET_BITS并不是LFS规范的要求 (仅提及该宏可作为指定数据类型off_t的长度的备选方法)。有些UNIX实现通过其他功能测试宏来获取同样的功能。

5.8.3 打印off_t值

　　数据类型off_t有时会定义为long long。因此，为了保证可移植性，在打印off_t值时，需要将其转换为long long，并使用格式化字符串%lld。当需要在不同的编译模块之间传递off_t值时，需要确保编译模块的设置统一，即都 (不) 使用宏_FILE_OFFSET_BITS。

　　同样的情况也适用于结构体stat中使用的数据类型blcnt_t。

5.9 目录/dev/fd

　　对于每个进程，内核都提供了特殊的虚拟目录/dev/fd。该目录下包含名称形如/dev/fd/n的文件，其中n是数字，对应进程的打开的文件描述符之一。例如，/dev/fd/0是进程的标准输入。

　　/dev/stdin、/dev/stdout和/dev/stderr分别是指向/dev/fd/0、/dev/fd/1和/dev/fd/2的符号链接。

　　/dev/fd实际上是指向特定于Linux的目录/proc/self/fd的符号链接。目录/proc/self/fd是特定于Linux的目录/proc/PID/fd的特例 (每个目录都包含与进程打开的所有文件相对应的符号链接)。

　　SUSv3并未说明目录/dev/fd，但多个UNIX实现都提供了该功能。

5.10 创建临时文件

　　在某些情况下，程序可能需要创建一些仅在程序运行时才会使用的临时文件，并且这些临时文件会在程序终止时删除。通常，临时文件在打开不久后就会取消链接/删除 (通过系统调用unlink())。

　　GNU C库提供了大量用于此目的的函数，这里仅介绍函数mkstemp()和tmpfile()。函数tmpnam()、tempnam()和mktemp()也能用于创建临时文件，但这些函数会产生安全漏洞，所以应该避免使用它们。

　　函数mkstemp()会根据参数template指定的模板生成唯一的文件名，并打开文件 (使用标志O_EXCL)。然后，返回文件描述符。对于生成的文件，所有者有读写权限，其他用户没有任何权限。

　　参数template的形式是路径名，并且最后6个字符必须是XXXXXX。这6个字符会被替换为唯一的文件名，并且修改后的字符串会存放在参数template中。因此，template必须是字符数组，而非字符串常量。
