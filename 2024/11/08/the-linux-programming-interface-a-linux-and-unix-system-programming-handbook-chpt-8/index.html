《Linux/UNIX系统编程手册》第8章 用户和组

发表于 2024-11-08 更新于 2024-12-17 分类于 计算机系统 本文字数： 11k

　　每个用户都有1个唯一的登录名 (用户名) 和1个关联的数字用户ID (即UID)。用户可以属于1个或多个组。每个组都有1个唯一的组名和1个组ID (即GID)。

　　用户ID和组ID的主要用途是确定各种系统资源的所有权和控制进程对系统资源的访问权限。

8.1 密码文件和影子密码文件

　　系统中的每个用户都在系统密码文件/etc/passwd中有1行记录，每行记录都包括以冒号:分隔的7个字段，如以下示例所示：

　　这些字段从左到右依次是登录名、加密后的密码、用户ID、组ID、注释、主目录和登录shell。

　　● 登录名就是登录时输入的用户名，也可以视为与用户ID关联的符号标识符。当通过ls等程序获取文件的所有者时，返回的就是登录名。

　　● 通常，UNIX会使用DES加密密码。在这种情况下，加密后的密码包括13个字符。当加密后的密码字段不是13个字符的字符串时，对应的登录名将会被禁用。若已启用影子密码 (通常如此)，则该字段会被忽略，此时，该字段通常会包含字符x (也可能出现其他未加密的字符)，真正的密码会存储在影子密码文件/etc/shadow中。若该字段为空，则对应的登录名无需密码就可以登录 (无论是否启用影子密码)。除了DES，还可以使用MD5等加密算法。MD5会输出128位的消息摘要 (即散列值)，该值会以34个字符的字符串形式存储在 (影子) 密码文件中。

　　● 若用户ID为0时，则表示对应的账户拥有超级用户特权，这样的账户通常只有1个，并且登录名是root。密码文件中可能会出现多行用户ID相同的记录，这允许用户使用不同的登录名来访问相同的资源，并且这些登录名可以属于不同的组。对于Linux 2.2及以下版本，用户ID存储为16位值 (即0~65536)。对于Linux 2.4及以上版本，用户ID存储为32位值。

　　● 组ID是用户所属的第1个组的ID。

　　● 注释是用户的相关信息，可以通过finger等程序获取。

　　● 主目录是用户登录后初始目录。用户登录后，主目录字段会成为环境变量HOME的值。

　　● 用户登录后，控制权会转移给登录shell，登录shell字段会成为环境变量SHELL的值。若该字段为空，则登录shell会默认为/bin/sh (即sh)。登录shell通常是shell，但可以是任何程序。

　　对于单机系统，所有密码信息都存储在/etc/passwd中。当使用网络信息系统或轻量级目录访问协议等系统来在网络环境中分发密码时，部分密码信息会存储在远程系统。只要应用程序使用getpwnam()和getpwuid()等函数来访问密码信息，NIS和LDAP的使用就对这些应用程序是透明的。

　　从历史角度，UNIX系统维护着所有用户信息，包括加密后的密码 (在/etc/passwd中)。因为各种非特权系统工具需要读密码文件中的信息，所以密码文件需要对所有用户都是可读的，这为密码破解程序创造了可乘之机 (密码破解文件会尝试对可能的密码进行加密，然后与密码文件中的密码进行匹配)。影子密码文件的设计初衷正是为了防止这类攻击，基本思想是将所有非敏感的用户数据存放在公开可读的密码文件，加密后的密码存放在仅对特权程序可读的影子密码文件。

　　除了登录名字段和加密后的密码字段，影子密码文件还包含很多安全相关的字段。

　　SUSv3没有说明影子密码，并且并非所有UNIX实现都提供了该功能。

8.2 组文件

　　用户所属的组由密码文件中的字段和组文件中的字段共同决定。对于早期UNIX实现，用户只能属于1个组。用户登录时的初始组由密码文件中的组ID字段决定，随后可以通过命令newgrp修改 (若对应的组设置了密码，则用户需要提供组密码)。4.2BSD允许用户能属于多个组，随后POSIX.1-1990将其标准化。

　　● 组名可以视为与组ID对应的符号标识符。

　　● 组密码是可选的，特权用户可以通过命令passwd来设置组密码。设置了组密码后，当不属于该组的用户启动属于该组的shell时，命令newgrp会要求用户输入组密码。若已启用影子密码，则该字段会被忽略，此时，该字段通常是字符x (也可能是其他字符串，甚至是空字符串)，真正的密码会存储在影子组文件/etc/gshadow (只有特权用户和程序能访问)。组密码的加密方式与用户密码的相似。

　　● 通常，只有1个组ID为0的组，并且该组的组名是root。对于Linux 2.2及以下版本，组ID存储为16位值 (即0~65536)。对于Linux 2.4及以上版本，组ID存储为32位值。

8.3 检索用户信息和组信息

8.3.1 从密码文件和组文件中检索记录

　　函数getpwnam()和getpwuid()可用于从密码文件中检索记录。

　　 函数getpwnam()会根据登录名 (即参数name) 检索密码文件，函数getpwuid()会根据用户ID (即参数uid) 检索密码文件。这2个函数都返回指向结构体passwd的指针。

　　SUSv3并未说明结构体passwd的成员pw_passwd和pw_gecos，但所有UNIX实现都提供了这2个成员。当且仅当未启用影子密码时，成员pw_passwd才包含有效信息。其他部分UNIX实现提供了该结构体的额外的非标准成员。

　　成员pw_gecos的名称源自早期UNIX实现，其中该成员存储用于与运行着通用电气综合运作系统的机器通信的信息。尽管这种用法早已过时，但该名称仍然存在。如今，该成员用于保存用户的相关信息。

　　函数getpwnam()和getpwuid()返回的指针指向的结构体是静态分配的，所以这2个函数不是可重入的，并且指针指向的结构体会被后续的getpwnam()、getpwuid()或getpwent()调用覆盖。

　　根据SUSv3，若密码文件中没有匹配的记录，则getpwnam()和getpwuid()会返NULL，并且不会设置errno。但是，在这一点上，很多UNIX实现并没有遵循SUSv3的规定。在这种情况下，除了返回NULL之外，这些UNIX实现还会将errno设置为非0值 (例如，ENOENT和ESRCH)。对于glibc 2.7之前的版本，会将errno设置为ENOENT，但从glibc 2.7开始，glibc遵循SUSv3的规定 (不设置errno)。这种给跨UNIX实现的差异部分是因为POSIX.1-1990不要求这些函数设置errno，并且允许它们根据情况设置合适的值。这一切造就了移植的不便。

　　函数getgrnam()和getgrgid()可用于从组文件中检索记录。

　　函数getgrnam()会根据组名 (即参数name) 检索组文件，函数getgrgid()会根据组ID (即参数gid) 检索组文件。这2个函数都返回指向结构体group的指针。

　　SUSv3并未说明结构体group的成员gr_passwd，但所有UNIX实现都提供了该成员。当且仅当未启用影子密码时，成员gr_passwd才包含有效信息。

　　与getpwnam()和getpwuid()类似，getgrnam()和getgrgid()返回的指针指向的结构体也会被后续的这2个函数的调用覆盖。

　　当getgrnam()和getgrgid()找不到匹配的记录时，它们的行为与getpwnam()和getpwuid()的类似。

　　SUSv3详细说明了一组与这4个函数等效的可重入函数，即getpwnam_r()、getpwuid_r()、getgrnam_r()和getgrgid_r()，这些函数的参数都包括1个结构体passwd (或group) 和1个用于存放结构体passwd (或group)中的成员指针指向的其他结构体的缓冲区。该缓冲区的大小可以通过调用sysconf(_SC_GETPW_R_SIZE_MAX)来获取 (对于组相关函数，则是调用sysconf(_SC_GETGR_R_SIZE_MAX))。

8.3.3 从影子密码文件中检索记录

　　函数getspnam()、getspent()、setspent()和endspent()用于从影子密码文件中检索单条记录或扫描所有记录。

　　这里不会介绍这些函数的细节，因为这些函数与对应的用于从密码文件中检索记录的函数类似。SUSv3没有说明这些函数，并且并非所有UNIX实现都提供了这些函数。

8.4 密码加密和用户认证

　　有些应用程序会要求用户进行身份认证，通常是输入用户名 (即登录名) 和密码。为此，应用程序可能会维护自己的用户名和密码数据库。有时，允许用户输入 (影子) 密码文件中定义的标准用户名和密码是必要或便利的。例如，ssh和ftp等用于登录到远程系统的网络应用程序验证用户名和密码的方式与标准登录程序的相同。

　　出于安全，UNIX系统使用单向加密算法来加密密码，这表示无法通过加密后的密码来获取密码明文。因此，验证输入的密码是否正确的方法就是将其以相同的算法加密，然后将结果与影子密码文件中的加密后的密码对比。加密算法封装在函数crypt()中。

　　函数crypt()会使用DES加密算法的变体来加密由最多8个字符组成的参数key (即密码)。参数salt是由2个字符组成的字符串，用于置乱 (使得加密后的密码更难破解)。该函数返回静态分配的由13个字符组成的字符串 (即加密后的密码)。

　　MD5算法生成的以美元符号$开头的由34个字符组成的字符串使得crypt()能够区分DES加密后的密码和MD5加密后的密码。

　　参数salt和crypt()返回的字符串中的字符都是从由64个字符组成的集合[a-zA-Z0-9/.]中选择。因此，对于同一密码明文，DES加密后的密码有4096种可能。

　　crypt()会将salt作为返回的字符串的前2个字符，这表示可以通过影子密码文件中已存在的加密后的密码来获取合适的salt值 (passwd等程序加密密码时会生成随机的salt值)。实际上，salt包含的字符可以多于2个，但函数crypt()会忽略salt的前2个字符之外的所有字符。

　　为了在Linux上使用crypt，编译程序时必须与库crypt一起链接 (使用选项-lcrypt)。

　　通常，通过函数getpass()来读取用户在终端输入的密码。

　　函数getpass()会先禁用终端特殊字符的回显和处理，然后打印参数prompt指定的字符串和读1行输入，最后将去除换行符后的以NULL结尾的输入作为结果返回 (该字符串是静态分配的，所以会被后续的getpass()调用覆盖)。在getpass()返回之前，会先将终端的设置恢复。

　　SUSv2详细说明了getpass() (将其标记为LEGACY)，但SUSv3没有说明该函数。大多数UNIX实现都提供了该函数。
